## Escalations and rulings

FF2 produced two HALT-LOUD escalations. Both were ruled by the owner on 2026-07-12 and the rulings shipped in the loop PRs.

### Escalation 1 - Q-W5-27: ficha field order not driven by template property order (jsonb)

- **Finding.** `form_templates.schema` is a jsonb column. Postgres normalizes jsonb object-key order (length, then bytewise), and the renderer plus in-ficha left nav both read `topLevelFields = Object.entries(schema.properties)` off that jsonb. Reordering the seed file's `properties` (the W5-27 loop's assumed mechanism) does NOT change render order. Consequence: production had rendered every osteopathy version in jsonb length-order, never the SPEC authoritative sequence, in any version. The file-order unit tests read the on-disk JSON and so masked the bug. Verified live on jaxmkwoxjcgzkwxgbayx. GREEN halted W5-27 before PR per the FF2 briefing.
- **Ruling (owner, 2026-07-12).** The x-order fix is approved. The v4 schema carries an `"x-order"` array (jsonb preserves array order) that is the FF2-A key sequence; `parseTemplateSchema` and `topLevelFields` sort by it when present, and it is the single order authority for both the form body and the left nav. Absent x-order falls back to current order, so v1/v2/v3 are never retrofitted (rule 5). The seeder `deepEqual` was made order-independent so it reports true file-DB parity. Migration-free. Scoped live UPDATE applied and verified; v4 stays active.
- **Standing decision.** x-order is the order authority for v4 and onward; v1/v2/v3 are never retrofitted. Nested-object orders (Outros grid, systems_review) remain jsonb-ordered and are out of W5-27 scope, flagged for a follow-up.
- Shipped in #559 (W5-27). Recorded: DECISIONS.md 2026-07-12 "W5-27 execution".

### Escalation 2 - Q-W5-33: final consent texts do not map cleanly to the consent structure

- **Finding.** The owner's three FF2 final texts (TEXT 1 treatment / TEXT 2 RGPD / TEXT 3 recording) did not line up with the ficha Consinto block, which had different items (`rgpd`, `sms`, `dataHandling`, each with `.v1/.v2/.v3` variants) plus PENDENTE-JP notices. TEXT 1 (treatment) had no consent item; `sms` and `dataHandling` had no owner text; the variant keys were obsolete once the wording was final. Resolving this is a consent-STRUCTURE change, conflicting with the loop's "keys frozen, only values change" clause. GREEN halted W5-33 before any edit.
- **Ruling (owner, 2026-07-12).** Consinto block = two items: `treatment` (new, TEXT 1 verbatim) + `rgpd` (TEXT 2 verbatim). Recording consent (TEXT 3 verbatim) lives on the Iniciar consulta step. The `sms` and `dataHandling` items, all nine `.v1/.v2/.v3` variant keys, and both PENDENTE-JP notices are REMOVED. SMS communication stays governed by the per-patient reminder opt-out flag (`patients.reminder_sms_enabled`, untouched). The keys-frozen clause was overridden for the CONSENT keys only; the twelve AI ingestion keys stay frozen (W5-13 compat 3/3). Signed-record guard: consent decisions read from stored `data._consent` and the deleted keys are no longer referenced by any code path, so no signed-record view breaks.
- **JP confirmation.** JP reviewed and confirmed the three consent texts on 2026-07-12.
- Shipped in #565 (W5-33). Recorded: DECISIONS.md 2026-07-13 "W5-33 execution".
