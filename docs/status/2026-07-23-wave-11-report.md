# Wave 11 — Separação de Produção — Close-out report (2026-07-23)

## What Wave 11 did (plain language)
The clinic app was running on the same Supabase project that had been the dev/build project since
the start. Wave 11 split that: it stood up a **clean new production database**, moved only the
clinic's configuration into it (no patient data — that stays behind on the old project), and
repointed the live app to it. The **old project is frozen and kept as an instant rollback.**

**The clinic is now live on the new production project.** Staff log in, the agenda shows the real
team and locations, and every page loads.

## What moved (W11-03)
- **11 configuration tables, 115 rows** copied old → new (clinics, staff, roles, services, prices,
  packs, schedules, forms, time-off). Verified exact + faithful (checked against the old data).
- **Zero patient data moved** — patients, appointments, clinical records, audit log all start empty
  on new. The old project keeps all of it, frozen.
- Auth logins for the 19 staff were copied too (addendum), so everyone keeps their existing password.

## Three incidents during go-live — all resolved
1. **No one could log in** — the migration copied staff records but not their login identities.
   Fixed by copying them.
2. **Login worked but data pages were empty/broke** — a copy bug stored three login-metadata fields
   in the wrong shape. Fixed in place, verified identical to the old data.
3. **All data pages failed** — the app's database address was the "direct" host, which no longer
   works from Vercel. Switched to the "pooler" host. **After this, the full Production smoke passed
   green** (login, dashboard, agenda with the real team + locations, admin pages, empty patient
   list; no errors; zero traffic to the old project).

## The old project (rollback) — safe
Never touched throughout the entire wave. Still frozen at its last state. Rollback = point the app
back at it. Retained as the rollback; a 30-day retention + eventual decision start once cutover is
declared final.
