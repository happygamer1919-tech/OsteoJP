# Staff platform a11y audit — 2026-06-19

**Scope:** `apps/web/app/` (all routes) and `apps/web/components/`. Source-code
audit only (no browser).

**Reference:** `docs/brand-tokens.md` for contrast tokens;
`.claude/agents/a11y-reviewer.md` for severity definitions.

**Severity definitions:**
- **P1 (blocker)** — Hard barrier: blocks a user from perceiving or operating the UI. Must be fixed before merge.
- **P2 (fix)** — Real a11y defect; not a hard blocker but must be corrected.
- **P3 (nit)** — Minor improvement; non-blocking.

**Not audited from source:** `@osteojp/ui` package components (`SidebarAppShell`,
`Drawer`, `Tabs`, `Combobox`, `Field`, `Input`, `Textarea`, `Button`, `Select`,
`Checkbox`, `Banner`, `Card`, `GlassPanel`, `GlassCard`, `StatusChip`,
`SegmentedControl`, `DatePicker`, `ToastProvider`). Their a11y guarantees are
assumed from the package contract. Skip link and primary navigation landmark
come from `SidebarAppShell` and are outside scope.

## Findings

### P1 — Blocker

**1. Body chart placement area is not keyboard-accessible**

- **File:** `apps/web/app/clinical/[id]/BodyChart.tsx:80`
- **Rule:** WCAG SC 2.1.1 Keyboard (Level A)
- **Description:** The click-to-place `<div onClick={place}>` at line 80 has no
  `role`, no `tabIndex`, and no `onKeyDown` handler. Keyboard users cannot
  place markers on the body chart at all. Any clinician using keyboard-only
  navigation (motor disability, switch access) is entirely blocked from the
  primary clinical input interaction. The read-only code path (`if (readOnly)
  return`) is irrelevant — the interaction is broken in the writable state.
- **Suggested fix:** Replace with `<button type="button">` (which receives
  keyboard focus naturally) or add `role="application" tabIndex={0}
  onKeyDown={handleKey}` where `handleKey` interprets arrow keys as relative
  coordinates and Enter/Space as a placement trigger. If a purely visual
  interface is acceptable, provide an alternative accessible list-based
  input (e.g. dropdowns for view × marker-type × approximate zone) alongside
  the canvas.

---

**2. PDF download error state not announced to screen readers**

- **File:** `apps/web/app/clinical/[id]/DownloadReportButton.tsx:40`
- **Rule:** WCAG SC 4.1.3 Status Messages (Level AA)
- **Description:** When the signed URL fetch fails, `setError(true)` is called
  and `<p className="text-xs text-error">` is injected into the DOM. The
  paragraph has no `role="alert"` or `aria-live` attribute. Screen-reader
  users have no way to know the download failed and may not discover the
  error paragraph if they do not re-read the entire region. Mirrors the
  portal's DownloadButton P1 finding exactly.
- **Suggested fix:** Add `role="alert"` to the error paragraph, or maintain
  a persistent `<p aria-live="assertive" aria-atomic="true">` region that
  receives the error string when `error` becomes true and is cleared on a
  new click attempt.

---

**3. Notas rápidas textarea has no accessible label**

- **File:** `apps/web/app/dashboard/notas-rapidas.tsx:29`
- **Rule:** WCAG SC 1.3.1 Info and Relationships / SC 4.1.2 Name, Role, Value (Level A)
- **Description:** The `<textarea>` at line 29 has no `<label>`, no
  `aria-label`, and no `aria-labelledby`. The only description hint is
  `placeholder={s["dashboard.notesPlaceholder"]}`, which disappears when the
  user types and is not reliably announced by all screen readers as a label.
  AT users cannot identify what this field is for. The parent `<GlassCard>`
  has a visible title (`dashboard.notes`) but it is not programmatically
  linked to the textarea.
- **Suggested fix:** Add `aria-label={s["dashboard.notes"]}` to the
  `<textarea>`, or wrap it in a `<label>` that references the GlassCard
  title text. Either approach associates the visible heading as the
  field's accessible name.

### P2 — Fix

**4. Patient form inputs suppress native focus outline without adequate replacement**

- **File:** `apps/web/app/patients/_components/patient-form.tsx:183`
- **Rule:** WCAG SC 2.4.11 Focus Appearance (Level AA, WCAG 2.2)
- **Description:** The shared `inputCls` string (used on every `<input>`,
  `<select>`, and `<textarea>` in the patient creation and edit forms)
  contains `outline-none focus:border-brand-teal`. `outline-none` removes
  the browser's native focus indicator. The only replacement is a border
  color change on `focus:` — a color-only change that does not meet the
  WCAG 2.4.11 requirement of a minimum 2 CSS pixel perimeter enclosing the
  component. The same pattern appears in the merge input in
  `patient-actions.tsx:87`.
- **Suggested fix:** Replace `focus:border-brand-teal` with
  `focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-focus-ring
  focus-visible:ring-offset-2` (the same pattern used by every
  `@osteojp/ui` primitive and the `admin-ui.ts` constants). Retain
  `focus:border-brand-teal` as a complementary visual cue if desired.

**5. Form submission errors appear without live-region announcement**

- **Files and lines:**
  - `apps/web/app/patients/_components/patient-form.tsx:156`
  - `apps/web/app/patients/_components/patient-actions.tsx:100`
  - `apps/web/app/auth/update-password/UpdatePasswordClient.tsx:257`
- **Rule:** WCAG SC 4.1.3 Status Messages (Level AA)
- **Description:** In all three components, the error state is set via
  `setError(...)` and rendered as a plain `<p>` with `text-error` styling.
  None of the paragraphs carry `role="alert"` or an `aria-live` attribute.
  When the paragraph is injected into the DOM after a failed submission or
  server action, screen readers do not announce it. Users relying on AT
  cannot tell whether their action succeeded or failed unless they manually
  re-read the form.
- **Suggested fix:** Add `role="alert"` (which implies `aria-live="assertive"
  aria-atomic="true"`) to each error `<p>`. Alternatively, hold a persistent
  `<p aria-live="assertive">` container that is always in the DOM and receives
  the error string.

**6. Clinical record save success not announced**

- **File:** `apps/web/app/clinical/[id]/RecordForm.tsx:82`
- **Rule:** WCAG SC 4.1.3 Status Messages (Level AA)
- **Description:** After the form saves successfully, `<p className="text-sm
  text-success">{s["clinical.saved"]}</p>` is rendered. There is no
  `role="status"` or `aria-live` on this paragraph, so screen readers are
  not notified of the save confirmation. A clinician saving a clinical record
  cannot confirm that the save succeeded without re-reading the page.
- **Suggested fix:** Add `role="status"` (which implies `aria-live="polite"`)
  to the success `<p>`. Polite is correct here — the announcement should not
  interrupt an in-progress screen-reader read.

---

**7. Attachment file-upload trigger has no visible focus indicator**

- **File:** `apps/web/app/clinical/[id]/Attachments.tsx:74`
- **Rule:** WCAG SC 2.4.7 Focus Visible (Level AA)
- **Description:** The file-upload trigger is a `<label>` wrapping a
  `className="hidden"` `<input type="file">`. When a keyboard user tabs to
  this control, focus lands on the hidden `<input>` (which has no visible
  focus ring because it is invisible), and the `<label>` that provides the
  visual affordance shows nothing. No `focus-visible:ring` or equivalent
  style is applied to the `<label>`. The attachment feature is completely
  inaccessible to keyboard users navigating by focus.
- **Suggested fix:** Style the `<label>` with a peer-focus ring driven by
  the hidden input: add `has-[:focus-visible]:ring-2 has-[:focus-visible]:ring-focus-ring
  has-[:focus-visible]:ring-offset-2` (Tailwind v4 `:has()` support), or
  restructure to use a `<button>` that triggers a ref-attached file input
  programmatically — the pattern used in the portal DownloadButton.

**8. Attachment upload error not announced**

- **File:** `apps/web/app/clinical/[id]/Attachments.tsx:79`
- **Rule:** WCAG SC 4.1.3 Status Messages (Level AA)
- **Description:** When the Supabase storage upload or URL-confirm fails,
  `setError(true)` triggers a `<p className="text-xs text-error">` that
  is conditionally rendered. No `role="alert"` or `aria-live` is present.
  A clinician who has just attached a file to a clinical record receives no
  programmatic notification that the upload failed.
- **Suggested fix:** Add `role="alert"` to the error `<p>`, following the
  same fix as finding #5.

**9. ReviewEditor buttons and messages lack focus rings and live regions**

- **File:** `apps/web/app/clinical/review/[recordId]/ReviewEditor.tsx:61,67,68,78`
- **Rule:** WCAG SC 2.4.7 Focus Visible (Level AA) and SC 4.1.3 Status Messages
- **Description:** Two issues in the same component:
  (a) The "Gravar" button (line 68) `className="rounded border px-3 py-1.5 text-sm font-medium"` and the "Finalizar" button (line 78) `className="rounded bg-accent-1-700 px-3 py-1.5 text-sm font-medium text-text-inverse"` have no `focus-visible:ring-2 focus-visible:ring-focus-ring` styling. Keyboard users have no visible focus indicator on either primary action in the AI review flow.
  (b) The error paragraph at line 61 and the success paragraph at line 67 are conditionally rendered without `role="alert"` / `role="status"`. Review save outcomes are not announced to screen readers.
- **Suggested fix:** Add `focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-focus-ring focus-visible:ring-offset-2` to both buttons. Add `role="alert"` to the error `<p>` and `role="status"` to the success `<p>`.

**10. Reschedule landing phone CTA link has no focus ring**

- **File:** `apps/web/app/r/[token]/page.tsx:128`
- **Rule:** WCAG SC 2.4.7 Focus Visible (Level AA)
- **Description:** The primary call-to-action on the public reschedule page is an `<a href="tel:...">` link styled `className="inline-block rounded bg-brand-teal px-4 py-2 font-medium text-text-inverse hover:bg-brand-teal/90"`. There is no `focus-visible:ring` on this element. The phone link is the only actionable control on the page for patients following an email reminder; its keyboard inaccessibility is visible to public unauthenticated users.
- **Suggested fix:** Add `focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-focus-ring focus-visible:ring-offset-2` to the `<a>` className.

**11. Episode detail page has no `<h1>`; heading hierarchy starts at `<h2>`**

- **File:** `apps/web/app/clinical/episodes/[id]/page.tsx:46`
- **Rule:** WCAG SC 1.3.1 Info and Relationships / best practice under SC 2.4.6 Headings and Labels
- **Description:** The episode detail page renders the episode title as `<h2>` (line 46) and the records section as `<h3>` (line 77). There is no `<h1>` on the page. Screen-reader users who navigate by heading level to orient themselves find no primary heading; the page title level is skipped. The next page in the patient flow (`/clinical/[id]`) correctly uses `<h1>`.
- **Suggested fix:** Change the episode title `<h2>` to `<h1>` and demote the records section heading from `<h3>` to `<h2>`. Alternatively, add a visually hidden `<h1>` containing the patient name (already available as `episode.patientName`) before the existing headings.

**12. Episode detail page links missing focus rings**

- **File:** `apps/web/app/clinical/episodes/[id]/page.tsx:39,67`
- **Rule:** WCAG SC 2.4.7 Focus Visible (Level AA)
- **Description:** The back link at line 39 (`className="inline-flex items-center gap-1 text-sm text-brand-teal"`) and the "Add record" link at line 67 (`className="rounded border border-brand-teal px-3 py-1.5 text-sm font-medium text-brand-teal hover:bg-surface-muted"`) have no `focus-visible:ring` styling. Both are keyboard-reachable but have no visible focus indicator in any browser where the default outline is suppressed by Tailwind's Preflight base styles.
- **Suggested fix:** Add `focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-focus-ring focus-visible:ring-offset-2` to both links, matching the `ghostLink` pattern used in `patients/[id]/page.tsx:30`.

**13. Body chart view toggle buttons have no `aria-pressed` state**

- **File:** `apps/web/app/clinical/[id]/BodyChart.tsx:52`
- **Rule:** WCAG SC 4.1.2 Name, Role, Value (Level A)
- **Description:** The four view-toggle buttons (Anterior, Posterior, Lateral esq., Lateral dir.) communicate the selected view only through CSS class changes (`bg-text-primary text-text-inverse`). There is no `aria-pressed` attribute. Screen readers cannot determine which view is currently active. A clinician navigating by keyboard has no programmatic indication of the selected view, and may place markers on the wrong anatomical surface.
- **Suggested fix:** Add `aria-pressed={view === v.value}` to each view button. Optionally wrap the group in a `role="group" aria-label={s["clinical.bodychartViews"]}` container.

### P3 — Nit

**14. Repeat icon carries `aria-label` without `aria-hidden` inside a text context**

- **Files and lines:**
  - `apps/web/app/agenda/agenda-grid.tsx:372` (inside `AppointmentBlock` button)
  - `apps/web/app/marcacoes/marcacoes-view.tsx:199` (inside `AppointmentRow` span)
- **Rule:** WCAG SC 1.1.1 Non-text Content / best practice
- **Description:** The `<Repeat>` icon (Lucide SVG) in both files is given
  `aria-label={s["appointment.recurring"]}` without `aria-hidden="true"`.
  The rendered SVG is treated as a named image by screen readers. Inside
  the appointment block button (agenda) or the patient name span (marcações),
  this produces an announcement like "Repetir, image, João Silva" — a
  semantic stutter that breaks the expected reading order. The `<User>` icon
  beside it is correctly `aria-hidden="true"`.
- **Suggested fix:** Change to `aria-hidden="true"` on the `<Repeat>` icon and
  convey the recurring status through a visually hidden `<span className="sr-only">{s["appointment.recurring"]}</span>` placed adjacent to the patient name, or include it in the button's accessible description via `aria-describedby`.

**15. Staff invite success message has no live-region announcement**

- **File:** `apps/web/app/admin/staff/StaffInviteForm.tsx:55`
- **Rule:** WCAG SC 4.1.3 Status Messages (Level AA)
- **Description:** On successful invite, a `<div>` containing the confirmation
  text and (for temp-password delivery) the provisional password is
  conditionally rendered. There is no `role="status"` or `aria-live` on the
  success div. An admin who has just sent an invite cannot confirm the outcome
  without re-reading the form. The error paragraph at line 72 uses
  `role="status"` (polite) rather than `role="alert"` (assertive) — for an
  error this is the wrong urgency level.
- **Suggested fix:** Add `role="status"` to the outer success `<div>` (lines
  55 and 61). Change `role="status"` on the error `<p>` (line 72) to
  `role="alert"` so the error is announced assertively.

**16. Appointment drawer recurring-scope radio group lacks `role="radiogroup"`**

- **File:** `apps/web/app/agenda/appointment-drawer.tsx:311`
- **Rule:** WCAG best practice; ARIA Authoring Practices Guide for radio groups
- **Description:** When editing a recurring appointment, the "Aplicar a"
  field (scope: one / following / series) renders three `<input type="radio">`
  elements inside a plain `<div>`. The radio group has no `role="radiogroup"`
  and no `aria-labelledby` linking it to the `<Field label="...">` heading.
  Screen readers may announce each radio as an isolated control rather than
  as members of a named group.
- **Suggested fix:** Wrap the `<div className="flex flex-col gap-1">` in
  `<fieldset>` with `<legend>{s["appointment.applyTo"]}</legend>`, or add
  `role="radiogroup"` and `aria-label={s["appointment.applyTo"]}` to the
  wrapping `<div>`.

**17. Admin services `<details>/<summary>` has no focus-visible ring**

- **File:** `apps/web/app/admin/services/page.tsx:129`
- **Rule:** WCAG SC 2.4.7 Focus Visible (Level AA)
- **Description:** The `<summary className="cursor-pointer ${adminHelp}">` used
  to expand location-price overrides for each service has no
  `focus-visible:ring` styling. `adminHelp` provides only text-size and color
  tokens. The `<summary>` element is natively keyboard-focusable; whether the
  browser's default `:focus-visible` outline renders depends on the Tailwind
  Preflight reset. The outline should be explicit, not browser-default-dependent.
- **Suggested fix:** Add `focus-visible:outline-none focus-visible:ring-2
  focus-visible:ring-focus-ring focus-visible:ring-offset-2 rounded` to the
  `<summary>` className.

**18. Episode records table headers missing `scope="col"`**

- **File:** `apps/web/app/clinical/episodes/[id]/page.tsx:86`
- **Rule:** WCAG SC 1.3.1 Info and Relationships (Level A)
- **Description:** The five `<th>` elements in the clinical records table
  (Template, Estado, Versão, Atualizado, [action]) have no `scope="col"`
  attribute. Without the scope, screen readers may announce cell values
  without their column header, particularly in complex table traversal modes.
  The action column header at line 90 is also empty with no `<span
  className="sr-only">` fallback.
- **Suggested fix:** Add `scope="col"` to all `<th>` elements. Add an
  `aria-label` or visually hidden `<span>` to the empty action header, e.g.
  `<th scope="col"><span className="sr-only">{s["clinical.open"]}</span></th>`.

**3 blockers, 10 fixes, 5 nits.** No failing text-color contrast was found: all
`text-v2-text-primary`, `text-v2-text-secondary`, `text-error`, `text-success-700`,
`text-warning-700`, `text-brand-teal`, and `text-text-inverse` usages fall on
backgrounds that clear AA at their respective font sizes. No `focus:outline-none`
suppression without a ring replacement was found outside of the `inputCls` string
documented in finding #4. No missing `alt` attributes or un-hidden decorative icon
omissions were found beyond the `<Repeat>` icon in finding #14. The login page,
dashboard navigation, agenda toolbar, patient list, and admin area all use the
`focus-visible:ring-focus-ring` pattern correctly; the defects are concentrated in
older pre-v2-restyle form components (`patient-form.tsx`, `patient-actions.tsx`)
and the clinical sub-pages (`BodyChart`, `Attachments`, `ReviewEditor`,
`episodes/[id]`). The `@osteojp/ui` package components could not be audited
from source; their a11y guarantees are assumed from the package contract.
