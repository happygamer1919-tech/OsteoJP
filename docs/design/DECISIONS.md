## 2026-07-01 - Append-only table conventions (choose deliberately)
- App-written append-only tables (e.g. analytics_events, 0025) use the POLICY pattern: SELECT+INSERT policies present, UPDATE/DELETE denied by absence-of-policy as 0 rows, table keeps the full DML grant. Mirrors audit_log. Deterministic 0-rows in both CI and dev.
- Admin-only add/remove tables (e.g. therapist_services, 0023) use the NO-GRANT pattern: the verb is revoked at the privilege layer, which THROWS SQLSTATE 42501. Tests against no-grant tables must assert the throw, not a 0-row result.
- Pick per table by intent; do not mix the two enforcement layers on one verb (the 0023-vs-0025 environment-split lesson).

## 2026-07-02 - Patient ID format ruling (JP) + implementation policy
- Ruling (JP, 2026-07-02): plain numbers only, NO prefix. Zero-padded display (e.g. `0001`). Migrated Fisiozero patients KEEP their original Fisiozero numbers; new patients get the next sequential number. Resolves the 2026-06-30 open question (QUESTIONS.md); no SPEC-patients conflict (SPEC-v2-patients specifies no numbering scheme).
- Implementation policy (locked for the 0029 migration):
  - Storage: `patients.patient_number integer`, stored UNPADDED (the integer `1`, not the string `0001`). Unique per tenant: `UNIQUE (tenant_id, patient_number)`. Padding is a DISPLAY concern only — zero-pad to a minimum of 4 digits at render (`0001`), never in the column; numbers past 4 digits render at their natural width (`12345`).
  - Scope: the number is per-tenant, not global — two tenants may hold the same `patient_number`. A cross-tenant duplicate is allowed and must be proven by test; a within-tenant duplicate is rejected by the unique constraint.
  - New patients: auto-assign `MAX(patient_number) + 1` within the tenant (first patient in a tenant = 1). Assignment is race-safe: it happens inside the tenant-scoped insert tx and relies on `UNIQUE (tenant_id, patient_number)` to reject a colliding concurrent insert, with bounded retry on unique-violation (or an equivalent per-tenant serialization such as a transaction-scoped advisory lock — mechanism chosen by the executor during recon, invariant fixed here: no duplicate, no gap-free guarantee required).
  - Fisiozero import: preserves each patient's ORIGINAL Fisiozero number as `patient_number`. Import reconciliation HALTs on ANY collision between an imported number and an already-assigned number within the same tenant; resolution is an OWNER decision made at import time, never auto-resolved by the importer. (Import path is future work; this policy binds it in advance.)
  - RLS/grants unchanged: `patient_number` is covered by the existing table-level `patients` grants and tenant RLS.
