## 2026-07-01 - Migration 0023 therapist-service mapping: mutability
- Created `therapist_services` (greenfield, SPEC-appointments.md §6): tenant-scoped join `therapist_user_id` (FK users) <-> `service_id` (FK services), with UNIQUE (tenant_id, therapist_user_id, service_id). users and services tables unchanged.
- Mutability: SPEC says only "admin-managed" and is silent on the CRUD verbs. This mirrors the append-only precedent (audit_log) for the "no in-place edit" shape while still allowing removal.

The "admin-managed" restriction is enforced server-side in the app layer, not in RLS — consistent with how role restrictions are handled elsewhere (RLS = tenant defense-in-depth; permission matrix = role enforcement).
- Column named `therapist_user_id` (not bare `user_id`): the FK targets a users row that must carry the therapist role, and the descriptive name reads clearly at the booking-flow call site. SPEC's "(user_id <-> service_id)" is the abstract relation, not a column-name mandate; no shape conflict.

## 2026-07-01 - Append-only table conventions (choose deliberately)
- App-written append-only tables (e.g. analytics_events, 0025) use the POLICY pattern: SELECT+INSERT policies present, UPDATE/DELETE denied by absence-of-policy as 0 rows, table keeps the full DML grant. Mirrors audit_log. Deterministic 0-rows in both CI and dev.
- Admin-only add/remove tables (e.g. therapist_services, 0023) use the NO-GRANT pattern: the verb is revoked at the privilege layer, which THROWS SQLSTATE 42501. Tests against no-grant tables must assert the throw, not a 0-row result.
- Pick per table by intent; do not mix the two enforcement layers on one verb (the 0023-vs-0025 environment-split lesson).

## 2026-07-02 - Patient ID format ruling (JP) + implementation policy
- Ruling (JP, 2026-07-02): plain numbers only, NO prefix. Zero-padded display (e.g. `0001`). Migrated Fisiozero patients KEEP their original Fisiozero numbers; new patients get the next sequential number. Resolves the 2026-06-30 open question (QUESTIONS.md); no SPEC-patients conflict (SPEC-v2-patients specifies no numbering scheme).
- Implementation policy (locked for the 0029 migration):
  - Storage: `patients.patient_number integer`, stored UNPADDED (the integer `1`, not the string `0001`). Unique per tenant: `UNIQUE (tenant_id, patient_number)`. Padding is a DISPLAY concern only — zero-pad to a minimum of 4 digits at render (`0001`), never in the column; numbers past 4 digits render at their natural width (`12345`).
  - Scope: the number is per-tenant, not global — two tenants may hold the same `patient_number`. A cross-tenant duplicate is allowed and must be proven by test; a within-tenant duplicate is rejected by the unique constraint.
  - New patients: auto-assign `MAX(patient_number) + 1` within the tenant (first patient in a tenant = 1). Assignment is race-safe: it happens inside the tenant-scoped insert tx and relies on `UNIQUE (tenant_id, patient_number)` to reject a colliding concurrent insert, with bounded retry on unique-violation (or an equivalent per-tenant serialization such as a transaction-scoped advisory lock — mechanism chosen by the executor during recon, invariant fixed here: no duplicate, no gap-free guarantee required).
  - Fisiozero import: preserves each patient's ORIGINAL Fisiozero number as `patient_number`. Import reconciliation HALTs on ANY collision between an imported number and an already-assigned number within the same tenant; resolution is an OWNER decision made at import time, never auto-resolved by the importer. (Import path is future work; this policy binds it in advance.)
  - RLS/grants unchanged: `patient_number` is covered by the existing table-level `patients` grants and tenant RLS.

## 2026-07-02 - 0029 patient_number: trigger auto-assign (executor deviation, owner-approved)
- Deviation from loop 0029 as written, APPROVED by owner (Ivan, 2026-07-02). Supersedes the loop file's Field 6 "single column + unique constraint" limit and Field 2 "app-layer-only assignment". Per owner instruction the committed loop file was left UNEDITED; the deviation is recorded here instead.
- Origin (the halt): pre-flight found `patient_number` cannot be `NOT NULL` with app-only assignment without redding the gated db suite. 16 patient-insert sites across 11 files insert without a number: 8 RLS test files (direct `insert into patients`), the Fisiozero import path `importPatient` (`packages/db/src/migration/upsert.ts:242`, exercised live by `migration-upsert-idempotency.test.ts`, which asserts `patients: 3` inserted), and 2 seeds. Every in-scope fix collided with a named loop constraint — a per-tenant auto-assign trigger hit Field 6, editing the import path hit Field 5 ("NO Fisiozero import code"), and threading the number through all 8 test inserts broke Field 5's "minimal `createPatient`-only wiring". Executor halted and surfaced the recommended default rather than silently picking one.
- Ruling (implemented): column `NOT NULL` + `UNIQUE (tenant_id, patient_number)`, backfill per tenant `1..N` ordered by `(created_at, id)` (soft-deleted rows included), plus a `BEFORE INSERT` trigger `public.assign_patient_number()` that sets `patient_number = MAX+1` per tenant ONLY when the inserted value IS NULL, serialized per tenant by a transaction-scoped advisory lock (`pg_advisory_xact_lock`) for race-safety. An EXPLICIT value passes through UNTOUCHED — so the "Fisiozero keeps original number" policy (ruling above) stays valid and future imports simply pass the original. `createPatient` still assigns `MAX+1` app-side under the same advisory lock; the trigger is the safety net for the other 15 insert paths.
- Backfill target corrected (owner amendment): live dev held 105 patients at execution, NOT the loop file's 50 (extra non-fixture rows exist on the shared dev project, separately under owner investigation). Binding DoD used: backfilled == `count(*)` at apply time, zero NULLs, per-tenant contiguity `1..N`. Evidence: 105 backfilled, 0 nulls, single dev tenant `3a2d0711-…` contiguous `1..105`; live new-patient round-trip assigned 106 and was purged; unique violation `23505` on `patients_tenant_number_uq`; cross-tenant `1`/`1` accepted.
- Drizzle schema note: `patient_number` is modelled `integer().notNull().default(sql\`null\`)`. The `sql\`null\`` is a type-level marker that keeps the column OPTIONAL in the Drizzle insert type, so the 15 non-setting paths compile untouched; the DB enforces `NOT NULL` and the trigger fills it. The column never stores a real NULL.

- Ruling (JP, 2026-07-02): patient notes require a FULL version history. Every edit to a patient note preserves the prior version, each version carrying its author and timestamp. Nothing is destructively overwritten — an edit appends a new version, and the full chain is retained and retrievable.
