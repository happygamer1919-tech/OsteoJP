# Portal a11y audit — 2026-06-17

**Scope:** `apps/portal/app/portal/` and `apps/portal/app/auth/` plus shared
components (`apps/portal/components/`). Source-code audit only (no browser).

**Reference:** `docs/brand-tokens.md` for contrast tokens;
`.claude/agents/a11y-reviewer.md` for severity definitions.

**Severity definitions:**
- **P1 (blocker)** — Hard barrier: blocks a user from perceiving or operating the UI. Must be fixed before merge.
- **P2 (fix)** — Real a11y defect; not a hard blocker but must be corrected.
- **P3 (nit)** — Minor improvement; non-blocking.

---

## Findings

### P1 — Blocker

**1. Download error state not announced to screen readers**

- **File:** `apps/portal/app/portal/documents/DownloadButton.tsx:11`
- **Rule:** WCAG SC 4.1.3 Status Messages (Level AA)
- **Description:** When a document download fails, `setError(true)` is called and the `Download` icon receives `className="text-error"`, changing its color to red. This is the only indication of failure. There is no `role="alert"`, `aria-live` region, or change to the button's accessible name. Screen-reader users have no way to know the download failed and cannot determine they should retry.
- **Suggested fix:** Update `aria-label` reactively to reflect the error state, e.g. `` aria-label={error ? `Erro ao descarregar ${fileName}. Tentar novamente` : `Descarregar ${fileName}`} ``. Alternatively, add a visually hidden live-region sibling that receives the error message when `error` becomes true.

### P2 — Fix

**2. NavButton renders a `<button>` for navigation links**

- **File:** `apps/portal/app/portal/dashboard/NavButton.tsx:14`
- **Affected usages:** `dashboard/page.tsx:85` ("Remarcar"), `dashboard/page.tsx:88` ("Detalhes"), `dashboard/page.tsx:116` ("Marcar consulta"), `appointments/AppointmentsView.tsx:78` ("Marcar consulta")
- **Rule:** WCAG SC 4.1.2 Name, Role, Value (Level A)
- **Description:** `NavButton` wraps `@osteojp/ui` `Button` and triggers `router.push(href)` via `onClick`. This renders as `role="button"` in the accessibility tree. Screen-reader users hear "Remarcar, button" — a button semantics implies an in-page action, not a page navigation. Users also cannot right-click to open in a new tab, and browser history handling differs from native links. The element IS keyboard-reachable (Tab + Enter), so it is not a hard keyboard block, but the role mismatch is a Level A failure.
- **Suggested fix:** Replace with `<Link href={href}>` styled via `buttonVariants()` from `@osteojp/ui`, or pass `href` directly to `Button` if the component exposes an `as` or `href` prop that renders an `<a>`. Navigation actions must have `role="link"`.

**3. Auth layout has no `<main>` landmark**

- **File:** `apps/portal/app/auth/layout.tsx:13`
- **Affected screens:** `/auth/login`, `/auth/activate`, `/auth/reset-password`
- **Rule:** WCAG SC 1.3.6 Identify Purpose / landmark best-practice (widely expected at AA)
- **Description:** The auth layout wraps children in a chain of `<div>` elements with no landmark regions (`<main>`, `<header>`, etc.). Screen-reader users relying on landmark navigation (a primary navigation strategy for many AT users) cannot jump to the form content. The portal section has a skip link and a `<main id="main-content">` (via `PortalChrome`), but auth pages have neither.
- **Suggested fix:** Wrap the `{children}` content in `<main>` inside `auth/layout.tsx`. Optionally add `<header>` around the `BrandLockup`. A skip link is not strictly required on single-landmark pages but should be added if the lockup header gains additional interactive content.

### P3 — Nit

**4. ChevronRight icons use `text-text-muted` — fails the 3:1 graphical-object threshold**

- **Files and lines:**
  - `apps/portal/app/portal/booking/BookingFlow.tsx:132` (location rows)
  - `apps/portal/app/portal/booking/BookingFlow.tsx:150` (service rows)
  - `apps/portal/app/portal/appointments/AppointmentsView.tsx:43` (appointment rows)
  - `apps/portal/app/portal/account/AccountView.tsx:32` (account rows)
  - `apps/portal/app/portal/account/AccountView.tsx:147` (password-change row)
- **Rule:** WCAG SC 1.4.11 Non-text Contrast (Level AA)
- **Description:** `text-text-muted` maps to `#8A98A6`, which measures approximately 2.9:1 against the white card background (`#FFFFFF`). WCAG SC 1.4.11 requires graphical objects to achieve at least 3:1. The chevrons are `aria-hidden="true"` (invisible to AT) and supplementary to the row's text label, so no sighted keyboard or AT user is blocked from operating the control — the row itself has clear interactive affordances. However, low-vision users relying on color alone to identify the affordance are affected.
- **Suggested fix:** Change `text-text-muted` to `text-text-secondary` (`#56697A`, ~5.5:1) on these chevrons, or to `neutral-600` (`#6E7E8D`, ~3.5:1) as a minimal-change option.

**5. Back link label "Voltar" carries no destination context**

- **File:** `apps/portal/app/portal/appointments/[id]/page.tsx:44`
- **Rule:** WCAG SC 2.4.6 Descriptive Labels / best practice under SC 2.4.4 Link Purpose
- **Description:** The back link in the appointment detail page reads "Voltar" with no accessible name or `aria-label` indicating the destination. A screen-reader user navigating the link list hears "Voltar, link" with no indication of where the link goes. The WCAG SC 2.4.4 link-in-context exemption is narrow; "Voltar" alone does not convey purpose.
- **Suggested fix:** Add `aria-label="Voltar para marcações"` to the link, or change the visible text to "Voltar para marcações" (which also aligns with the brand-voice §5 forward/back link convention).

---

**6. Disabled cancel button removed from tab order**

- **File:** `apps/portal/app/portal/appointments/[id]/AppointmentActions.tsx:62`
- **Rule:** Best practice; WCAG SC 2.1.1 Keyboard (perceivability of disabled states)
- **Description:** When an appointment is within the 24-hour cutoff, the cancel button uses the native HTML `disabled` attribute, which removes the element from the tab order entirely. Keyboard users tabbing through the detail screen cannot discover that a cancel action exists (even in a disabled state). The explanatory paragraph below (`Não é possível cancelar…`) IS visible to sequential AT readers, but the button itself is unreachable via Tab.
- **Suggested fix:** Replace `disabled={!outsideCutoff}` with `aria-disabled={!outsideCutoff}` and guard the click handler in JavaScript (`if (!outsideCutoff) return`). This keeps the button in the tab order, allows screen readers to announce it as "Cancelar consulta, dimmed, button", and lets the subsequent paragraph provide the reason.

---

**7. Password requirement indicators have no live region**

- **File:** `apps/portal/app/auth/activate/page.tsx:124`
- **Rule:** WCAG SC 4.1.3 Status Messages / best practice for dynamic form feedback
- **Description:** The `<ul>` at line 124 renders two `Requirement` list items that change color and icon (`Check` vs `Circle`) as the user types. These visual state changes are not communicated to screen readers: there is no `aria-live` region, `role="status"`, or `aria-atomic` attribute on the list or its container. A user relying on a screen reader types their password and receives no feedback about whether requirements have been met.
- **Suggested fix:** Add `aria-live="polite"` to the `<ul className="flex flex-col gap-1 text-xs">`. Also add visually hidden text to each `Requirement` item that makes the met/unmet state explicit (e.g. prepend a `<span className="sr-only">{met ? 'Concluído: ' : 'Pendente: '}</span>` before the children) so the announcement is informative rather than just re-reading the label.

---

**8. Skip link uses `focus:` utilities and no `ring-focus-ring` token**

- **File:** `apps/portal/components/layout/PortalChrome.tsx:57`
- **Rule:** Design-system consistency; `docs/brand-tokens.md §1.9`
- **Description:** The skip link uses `focus:not-sr-only` / `focus:absolute` (unconditional focus, not `focus-visible:`) and does not apply `ring-focus-ring`. When focused via keyboard, the browser's default outline renders alongside the revealed teal background — visually functional but inconsistent with the `ring-2 ring-focus-ring ring-offset-2` pattern used on every other interactive element in the portal.
- **Suggested fix:** Add `focus-visible:ring-2 focus-visible:ring-focus-ring focus-visible:ring-offset-2` to the skip link className. The `focus:not-sr-only` pattern is intentionally unconditional (skip links should appear even on pointer focus) and can be retained alongside the `focus-visible:ring` addition.

**1 blocker, 2 fixes, 5 nits.** No failing text-color contrast was found: all
`text-primary`, `text-secondary`, `text-error`, `text-success-700`,
`text-warning-700`, and `text-accent-2-700` usages are within AA limits for
their respective font sizes. No missing `alt` attributes or `aria-hidden`
omissions were found on directly visible icon usages. No `focus:outline-none`
suppression without a ring replacement was found. The `@osteojp/ui` package
components (`Button`, `Field`, `Input`, `Dialog`, `Drawer`, `Switch`, `Card`,
`StatusChip`, `SegmentedControl`, `PortalShell`) could not be audited from
source; their a11y guarantees are assumed from the package contract but not
verified here.
