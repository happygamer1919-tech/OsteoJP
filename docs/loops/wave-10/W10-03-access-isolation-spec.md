## Field 1. Scope and ground truth

Author `docs/design/SPEC-isolation.md`: a per-person isolation spec so a therapist logs in and sees ONLY themselves, their own patients, their own location, with NO location or therapist switching; only admin and owner keep cross-visibility. Ground truth (recon at authoring 2026-07-20, embed - the executor verifies read-only against `packages/auth` + the routes, executor runs with ZERO memory; all file:line refs are the starting map):

- **Role model (`packages/auth/permissions.ts:10`):** `ROLES = ["owner", "admin", "therapist", "reception"]` - note the fourth role is **`reception`** (not "receptionist"/"staff"). **`owner` is distinct from `admin`** (different capability sets; only `owner` may grant the `owner` role, `permissions.ts:176,187`). The JWT app-role claim is **`user_role`** (a text slug; `role` is reserved by Supabase for `SET ROLE`), values exactly `owner|admin|therapist|reception`, stamped by the access-token hook from `roles.slug` (`0002_auth_token_hook.sql:53-65`). Patients are a SEPARATE trust domain (`role = 'patient'`, `packages/auth/patient.ts`), not a staff `Role`.
- **Capability matrix already exists as DATA (`packages/auth/permissions.ts:89-154`):** `PERMISSIONS: Record<Role, ReadonlySet<Capability>>` over 25 capabilities (`permissions.ts:26-58`: `patients:read/write/delete/recover`, `appointments:read/write/delete`, `services:*`, `locations:*`, `clinical_records:read/author/review/sign`, `invoices:read/issue/void`, `users:read/manage`, `roles:read/manage`, `settings:read/manage`, `audit_log:read`, `statistics:read`). Checked via `can(role, capability)` (`permissions.ts:156`) and enforced by `assertCan(role, capability)` throwing `ForbiddenError` (`packages/auth/guard.ts:27-31`). **This is a capability grid, not a data-scope model** - it says which VERBS a role may perform, not WHICH ROWS it may see. Per-person isolation is a data-scope layer ON TOP of this grid.
  - owner = ALL. admin = full operational + `clinical_records:read` (no authoring), no `roles:manage`/`patients:recover`/`statistics:read`. therapist = `patients:read/write`, `appointments:read/write`, `clinical_records:read/author/review/sign`, `services:read`, `locations:read`, `invoices:read` (no delete/settings/user-admin/statistics). reception = scheduling + `invoices:issue`, NO `clinical_records` at all.
- **tenant_id + role come from the JWT, fail-closed:** `getRequestContext()` (`apps/web/lib/auth/context.ts:24-40`) reads `{ tenant_id, user_role, sub }` from the VERIFIED claims and returns `null` on any missing/invalid value; `runScoped` -> `withTenantContext` sets the claims as a bound parameter for RLS (`packages/db/src/client.ts:115-127`). **tenant_id is NEVER read from a query param or body.** This is the invariant the spec keeps: nothing widens tenant scope, tenant_id stays JWT-only.
- **Therapist self-scope TODAY is appointment-practitioner-only, and only on two surfaces:** `lockTherapist = actor.role === "therapist"` in the agenda (`apps/web/app/agenda/page.tsx:58`) and marcacoes (`apps/web/app/marcacoes/page.tsx:94`) forces `practitionerId = actor.userId` into `listAppointments` (`apps/web/lib/scheduling/data.ts:144-164`). **Everything else is tenant-wide for a therapist:** the patients list (`apps/web/lib/patients/queries.ts:100-152`) is NOT therapist-scoped; clinical records RLS grants a therapist read across the WHOLE tenant with an explicit standing TODO (`0001_rls.sql:175-176`: "therapist -> all in-tenant. TODO v0.1: tighten to patients-they-treat once user_locations / appointment scoping exists"). So the spec is FILLING this exact gap.
- **"Their patients" has NO data model (critical):** there is no `assigned_therapist`/`primary_therapist`/`treating` column on `patients` (`packages/db/src/schema.ts:431-490`; the only staff FK is `created_by`, provenance not assignment). "Patients a therapist may see" must be DERIVED - the natural derivation is "patients with an appointment where `practitioner_id = actor.userId`" (index `appointments_practitioner_start_idx` on `(practitioner_id, starts_at)` already exists, `schema.ts:609`; secondary `practitioner_2_id` exists too if co-treatment counts). Whether to derive-from-appointments or introduce an explicit assignment column is an OPEN QUESTION (Field 1 open list). Do NOT assume a column exists.
- **A therapist's location is DERIVED from `availability_templates`** (`getTherapistLocationIds` / `listTherapistLocationAssignments`, `apps/web/lib/scheduling/therapist-locations.ts`) - there is no `therapist_locations` table. "Their own location" reuses this derivation.
- **RLS posture:** fail-closed, tenant-keyed on `jwt_tenant_id()` (`0001_rls.sql:30-45`); all 13 domain tables have RLS; the ONLY role-differentiated table is `clinical_records` (still tenant-wide); the ONLY per-user (`auth.uid()`) policy anywhere is `quick_notes_own_row` (`0018_quick_notes.sql:44-48`). **Per-therapist RLS on patients/appointments/records is NET-NEW** - the spec's enforcement plan states this and keeps RLS fail-closed (nothing widened).
- **Selectors:** rendered in `apps/web/app/agenda/agenda-view.tsx`. The **therapist** selector ("Todos os terapeutas", `:194-209`) is ALREADY role-gated (`{!lockTherapist && ...}`). The **location** selector ("Todas as localizacoes", `:210-231`) is **NOT** role-gated - a therapist still sees and operates it today. "Therapist role loses location and therapist selectors entirely" therefore needs a NEW gate on the location Select (the therapist Select is already handled). Same `lockTherapist` prop exists in marcacoes (`marcacoes-view.tsx:415`).
- **Surfaces (routes, `apps/web/app/...`):** agenda (`agenda/page.tsx`), marcacoes (`marcacoes/page.tsx`), patients list (`patients/page.tsx`), patient detail (`patients/[id]/`), fichas/clinical (`clinical/`), estatisticas (`estatisticas/` + `painel` + `indicadores`, owner-only `statistics:read`), equipa/staff (`admin/staff/`), servicos (`admin/services/`), locations (`admin/locations/`), settings (`admin/settings/`), pacientes-eliminados (owner-only `patients:recover`), invoicing (`invoicing/`), acoes destrutivas (the password-gated block on `patients/[id]`), perfil (`perfil/`). The **portal** is the separate patient-facing app (`apps/portal`, a different trust domain) - staff isolation does not change it; the spec states the portal is out of scope for the staff matrix and unaffected.

**Scope:** a committed spec `docs/design/SPEC-isolation.md` with (1) the role model (owner/admin/therapist/reception, grounded in `packages/auth`); (2) a capability/visibility MATRIX of surfaces x roles (agenda, patients list, patient detail, fichas, marcacoes, estatisticas, equipa, servicos, acoes destrutivas, portal) marking for each role: no-access / own-only / location-scoped / tenant-wide; (3) the enforcement plan, SERVER-SIDE FIRST (query scoping by location + therapist identity, RLS stays fail-closed, tenant_id from JWT only, nothing widened); (4) the UI consequences (therapist loses the location AND therapist selectors entirely); (5) explicit OPEN QUESTIONS for owner ruling. Read-only inspection; NO code, NO migration, NO schema. The only writes are the spec doc + the BACKLOG row flip + the QUESTIONS entries.

**OPEN QUESTIONS the spec MUST list (at minimum) for owner ruling:**
- **Rodica's role class** - which of owner/admin/therapist/reception does Rodica hold, and does the isolation apply to her?
- **What "their patients" means for a therapist** - patients with ANY appointment with them (derive from `appointments.practitioner_id`, incl. secondary `practitioner_2_id`?) vs an explicit assignment column (net-new schema).
- **Whether reception sees all therapists at their own location** - reception currently sees the whole tenant; is reception location-scoped, therapist-agnostic, or unchanged?
- **Coverage / substitution scenarios** - a therapist covering a colleague (seeing that colleague's patient once they have an appointment with the covering therapist?); does treating a patient once grant standing visibility?
- **Whether admins are location-scoped or global** - admin keeps cross-visibility, but is admin visibility tenant-wide across all locations or scoped to their location(s)?
- **Owner vs admin split for cross-visibility** - confirm both owner and admin keep full cross-visibility (the brief says "only admin and owner keep cross-visibility").
- **Does isolation change what a therapist sees in estatisticas** - estatisticas is already owner-only `statistics:read`, so therapists have no access; confirm no change is wanted there.
