State for each whether it is purged (patient-linked test data) or retained.

- **FK / back-pointer map (embed, executor confirms):** **every FK in this schema is `ON DELETE NO ACTION ON UPDATE NO ACTION` EXCEPT `tenant_id` (which is `ON DELETE CASCADE`).** None of the patient/appointment/clinical_record back-pointers CASCADE or SET NULL. **Consequence for the cleanup: a purge cannot rely on cascade - it must delete children BEFORE parents, bottom-up.** Back-pointers to map:
  - **-> `patients.id`:** `appointments.patient_id` + `appointments.patient_2_id`, `patient_locations.patient_id`, `clinical_episodes.patient_id`, `clinical_records.patient_id`, `attachments.patient_id`, `appointment_notes.patient_id`, `patient_note_revisions.patient_id`, `patient_pack_instances.patient_id`, `invoices.patient_id`, `patient_form_submissions.patient_id`, `analytics_events.patient_id`.
  - **-> `appointments.id`:** `appointment_notes.appointment_id`, `clinical_records.appointment_id`, `invoices.appointment_id`. (Plus in-row bare-uuid pointers with NO FK constraint: `recurrence_parent_id`, `booking_group_id`, `batch_id`.)
  - **-> `clinical_records.id`:** `clinical_records.supersedes_id` (self-FK addendum chain, `schema.ts:777`), `attachments.clinical_record_id`, `record_annulments.record_id`, `ai_ingestion_requests.clinical_record_id`, `patient_form_submissions.clinical_record_id`.

- **The immutability mechanism (embed, executor confirms EXACTLY):** trigger **`clinical_records_enforce_immutability`**, `BEFORE UPDATE OR DELETE ON public.clinical_records FOR EACH ROW`, function `public.enforce_clinical_record_immutability()` - defined `packages/db/migrations/0001_rls.sql:232-255`, redefined `0005_patient_merge_multilocation.sql:56-91`. **It blocks BOTH UPDATE and DELETE when the existing row `status IN ('locked','signed')`, raising `check_violation`; a `draft` (including AI-pending) record is freely updatable/deletable.** It runs in the BEFORE phase with no BYPASSRLS escape, so **it applies EVEN to `service_role`** (`0001_rls.sql:225-230`). Determine precisely, against the live DB, whether any `locked`/`signed`/annulled rows exist: those CANNOT be deleted by ANY path and are the BLOCKED residue.

- **Sanctioned delete paths (embed, executor confirms coverage):** all four hard-delete paths share the scrypt gate `verifyDeletePassword(ctx, password)` (`apps/web/lib/admin/appointment-delete-password.ts:23`; hash in `tenants.settings.secrets.appointmentDeletePasswordHash`). **Every app-layer hard-delete is reference-guarded** - it refuses when protected downstream data exists - so the app paths alone CANNOT bulk-purge a patient that has appointments/records:
  - **`hardDeletePatient(id, password)`** (`apps/web/lib/patients/actions.ts:246`, cap `settings:manage`): BLOCKED if ANY `clinical_records` row (`has_clinical_records`) or any other reference (episodes, appointments, notes, note revisions, invoices, attachments, form submissions, analytics, merge-losers) exists; only deletes the `patient_locations` junction then the `patients` row.
  - **`hardDeleteAppointment(id, password)`** (`apps/web/lib/scheduling/actions.ts:881`, cap `settings:manage`): BLOCKED (`linked_records`) if any note/clinical record/invoice references it; else deletes child analytics rows then the appointment.
  - **`hardDeleteClinicalRecord(ctx, id)`** (`apps/web/lib/clinical/records.ts:575`, cap `clinical_records:author`, password-gated via `hardDeleteRecordAction`): **DRAFT only** (`records.ts:586`); AI-pending IS `status = 'draft'` so it IS deletable; `locked`/`signed` blocked at both app layer and the DB trigger.
  - **`deleteStaffMember`** (`apps/web/lib/admin/staff.ts:467`): staff are REAL and STAY; do not delete users in this wave. Documented here only for boundary completeness.
  - **Soft-delete + restore:** `softDeletePatient` (`deleted_at = now()`), `restorePatient` (`deleted_at = null`, refuses merge-losers), `listDeletedPatients` (reads `deleted_at IS NOT NULL OR merged_into_id IS NOT NULL`) - the "Pacientes eliminados" surface. Soft-deleted patients still carry all child rows (NO ACTION FKs), so the purge must handle them too.
